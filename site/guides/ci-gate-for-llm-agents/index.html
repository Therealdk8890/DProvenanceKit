Gate your CI when your agent's reasoning drifts

Most LLM testing in CI/CD stops at unit tests, and unit tests can't see that a prompt tweak made your agent skip its verify step. This guide adds the missing stage: a reasoning-regression gate that diffs each agent run against a golden baseline and fails the pull request when the execution path drifts — a dropped tool call, a looping step, a changed path.

Everything below is open source (Apache-2.0), runs on local SQLite files, and needs no hosted backend. New to golden traces? Start with the concepts guide.

# 01 · the missing pipeline stage

Dashboards tell you after you shipped. A gate blocks the merge before.

AI agent observability and AI agent monitoring answer one question: what did the agent do in production? That's necessary — and structurally too late. By the time a dashboard shows the regression, the PR that caused it is merged and deployed, and you're bisecting commits while users see the broken behavior. A tracing dashboard is a rear-view mirror; nothing in it stops the next bad merge.

Deterministic code solved this decades ago with a pipeline stage: tests run on every pull request, and a failure blocks the merge. Agents deserve the same stage. LLM CI/CD quality gates work exactly like unit tests, just on a different artifact — instead of asserting on return values, you assert on the execution path: which steps ran, in what order, through which tools. Record a known-good run once, then diff every candidate run against it. If the paths match, merge. If a critical step vanished, exit code 1.

That's the whole loop AI agent testing in CI needs — and the last two stages are the ones most stacks are missing (evaluating hosted platforms instead? See DProvenanceKit vs LangSmith):

run→ record→ query→ diff→ detect regressions→ gate in CI

# 02 · the server-less gate

One CLI command. No service to stand up.

The gate is a single command: dprovenancekit gate. Traces are plain local SQLite files that live in your repo or as CI artifacts — there is no backend, no account, no API key, and nothing leaves the runner. Point it at a database, name the golden and candidate context ids (it selects the newest run recorded under each), and it exits non-zero on regression. Any CI system that understands exit codes can gate on it.

The fingerprint line is a cheap first signal: a run fingerprint is a structural identity of the execution path, so a reordered tool, a skipped retrieval step, or a dropped verify step each change it. The per-step diff and the engine's severity assessment explain why it changed.

# 03 · github actions

The workflow: copy, paste, gate.

For GitHub, the DProvenanceKit Regression Gate action on the Marketplace wraps the CLI. On regression it fails the job and posts a sticky PR comment with the diff, so reviewers see exactly which steps changed without opening logs. The full workflow:

.github/workflows/reasoning-regression.ymlyaml
name: reasoning-regression
on: pull_request

permissions:
  contents: read
  pull-requests: write   # required for the PR comment

jobs:
  gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # Produce a SQLite trace database with your golden + candidate runs.
      - name: Record traces
        run: python scripts/record_traces.py   # your script; writes traces.sqlite

      - name: Regression gate
        uses: Therealdk8890/dprovenancekit-action@v1
        with:
          db-path: traces.sqlite
          golden-context: golden        # newest run recorded with this context id
          candidate-context: candidate
          max-level: none               # strict: any divergence fails

The "Record traces" step is yours: run your agent once against the committed golden baseline and once with the PR's code, writing both into traces.sqlite (or commit the golden database and only record the candidate — see golden-run hygiene below). More context on the gate lives at dprovenance.dev/#ci.

# 04 · what a failure looks like

Unit tests passed. The reasoning gate didn't.

That's the point of the extra check: your unit tests assert the code still returns the right shapes, and they'll happily stay green while the agent's plan quietly loses its verification step. On the PR, the two checks disagree — and the sticky comment shows the reviewer the removed step, not a stack trace:

✓tests (pull_request)Successful in 41s
✗reasoning-regression (pull_request)Failing after 12s
Mockup of GitHub's PR checks UI — these rows and their durations are illustrative, not a screenshot of a real CI run. The gate output behind the failing row is the real transcript in section 02; watch it replayed on the homepage.

The failing check reads like a code review comment about reasoning: removed: verify.claims, added: search.web ×2. The author either fixes the drift or argues — in the PR, before the merge — that the new path is intentional and re-records the baseline.

# 05 · gitlab ci

GitLab: include one template.

The same gate ships as a GitLab CI template in the repo's gitlab/ directory, with parity to the GitHub Action: it runs on merge request pipelines, fails the MR job on regression, and can post the diff as an MR note.

.gitlab-ci.ymlyaml
include:
  - remote: 'https://raw.githubusercontent.com/Therealdk8890/DProvenanceKitPython/main/gitlab/dprovenancekit.gitlab-ci.yml'
  • Required variables: DPROV_DB (path to the SQLite trace database), DPROV_GOLDEN and DPROV_CANDIDATE (the golden and candidate run ids).
  • Optional: DPROV_MAX_LEVEL (none|low|medium|high, default none), DPROV_ALLOW_DIVERGENT, and separate DPROV_GOLDEN_DB/DPROV_CANDIDATE_DB paths.
  • MR note: set DPROV_GITLAB_TOKEN to a token with api scope (the default CI_JOB_TOKEN can't post notes).

# 06 · tuning

Strict by default. Loosen it deliberately, not accidentally.

A gate that cries wolf gets deleted within a month, so the policy knobs matter. The gate has two independent dials: a severity threshold (the engine rates each regression none/low/medium/high — a removed critical step like verify.claims is high) and per-step strictness (by default any removed, added, reordered, or ambiguous step fails, regardless of severity). The same policy is available in Python for tests and custom harnesses:

gate_policy.pypython
from dprovenancekit import AlignmentProfile, AnyEquivalenceEvaluator, RegressionLevel
from dprovenancekit.testing import assert_no_regression

# Strict default: every step must align exactly, severity must be none.
assert_no_regression(golden=golden_run, candidate=candidate_run)

# Looser: gate on severity only, tolerate benign per-step changes.
assert_no_regression(
    golden=golden_run, candidate=candidate_run,
    max_regression_level=RegressionLevel.HIGH,
    allow_divergent_steps=True,
)

# Ignore volatile payload fields (timestamps, token counts) with your own
# evaluator — strip_volatile is your function.
stable = AnyEquivalenceEvaluator(
    evaluator_identifier="IgnoreVolatile_v1",
    evaluator=lambda a, b: 1.0 if strip_volatile(a) == strip_volatile(b) else 0.0,
)
assert_no_regression(golden=golden_run, candidate=candidate_run, evaluator=stable)

# Guard step ORDER too: reorder detection needs a span-aware profile.
assert_no_regression(
    golden=golden_run, candidate=candidate_run,
    profile=AlignmentProfile.developer_debug_v1,
)
  • max-level / max_regression_level — the worst severity that still passes. Keep none for agents where every step is load-bearing; raise it when low-stakes steps churn.
  • allow_divergent_steps — tolerate benign per-step changes and let severity alone decide. Useful when the agent legitimately varies (an optional cache-lookup step, say) but critical steps must never disappear.
  • Custom evaluators — the default requires payloads fully equal; volatile fields (timestamps, token counts, latencies) belong in an evaluator that ignores them, not in a looser gate.
  • Reordering — under the default linear profile, a pure reorder binds 1:1 and reads as still-matching. If order is part of what you're guarding, pass the span-aware AlignmentProfile.developer_debug_v1; a reordered critical step is then a high-severity regression, same as a removed one.
  • No exceptions wanted? RegressionGate(...).check(golden, candidate) returns a RegressionReport verdict without raising, for custom reporting.

Independent of any golden run, absolute anomaly rules catch the two classic agent failure modes — a required step that vanished, and a tool stuck in a loop — from a JSON rule registry:

.github/dprov-rules.jsonjson
{
  "rules": [
    { "type": "tool_drop", "required_step": "safety_check" },
    { "type": "looping", "step": "web_search", "max_repeats": 5 }
  ]
}

$ dprovenancekit anomalies --db traces.sqlite --rules rules.json — runs the same rules locally or on every PR, no golden baseline required.

# 07 · golden-run hygiene

Treat goldens like snapshots, because they are.

A golden trace is a snapshot of your agent's reasoning, and the workflow is the one you already know from snapshot testing. The pytest plugin makes the whole lifecycle two commands:

test_agent.pypython
def test_research_agent(golden_trace):
    with golden_trace("research-agent"):
        run_my_agent()
shellsh
$ pytest --dprov-update-golden   # record (or intentionally update) the baseline, then commit it
$ pytest                         # every run after gates against tests/goldens/research-agent.sqlite
  • Commit goldens to the repo. They're small, deterministic SQLite files under tests/goldens/ (configurable via the dprov_golden_dir ini option) — and because the gate itself can diff them, a baseline update in a PR is reviewable, not opaque.
  • Re-baseline only on intentional behavior change. When the agent is supposed to reason differently, run pytest --dprov-update-golden (or re-record your golden context) and review the golden diff in the PR like any snapshot update. If you're updating the baseline to silence a failure you can't explain, that's the regression.
  • Tune per test, not globally. Options pass through the fixture: golden_trace("research-agent", max_regression_level="high") — the same knobs as section 06.
  • Mixed stacks stay comparable. Trace Specification v1 (the repo's conformance/ directory, frozen golden vectors) keeps the Python SDK and the original Swift SDK behaviorally equivalent — a golden recorded against one implementation stays meaningful across a Swift-and-Python codebase.

One more hygiene point: trust the detector before you gate on it. The regression-detection benchmark ships with the library — dprovenancekit evaluate reports precision 1.000 / recall 1.000 / F1 1.000 across 8 standard and 5 adversarial scenarios, so you can re-verify the corpus on your own machine instead of taking a README's word for it.
